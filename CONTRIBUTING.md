# Contributing to Bonnie's Invoice Manager

Thank you for your interest in contributing to this project! This document provides guidelines for contributing.

## ü§ù How to Contribute

### Reporting Issues

If you find a bug or have a suggestion:

1. **Search existing issues** to avoid duplicates
2. **Create a new issue** with:
   - Clear, descriptive title
   - Detailed description of the issue/suggestion
   - Steps to reproduce (for bugs)
   - Expected vs actual behavior
   - Screenshots if applicable
   - Your environment (browser, OS)

### Suggesting Enhancements

Enhancement suggestions are welcome! Please:

1. Check if the enhancement has already been suggested
2. Explain why this enhancement would be useful
3. Provide examples of how it would work
4. Consider backward compatibility

### Code Contributions

#### Before You Start

1. Fork the repository
2. Create a new branch: `git checkout -b feature/your-feature-name`
3. Make your changes
4. Test thoroughly
5. Commit with clear messages
6. Push to your fork
7. Create a Pull Request

#### Development Setup

```bash
# Clone your fork
git clone https://github.com/YOUR_USERNAME/bonnies-invoice-manager.git

# Create feature branch
git checkout -b feature/my-new-feature

# Make changes and test using preview.html

# Commit changes
git add .
git commit -m "feat: Add new feature description"

# Push to your fork
git push origin feature/my-new-feature
```

#### Code Style Guidelines

- **JavaScript**: Use ES6+ syntax where appropriate
- **HTML**: Semantic HTML5 elements
- **CSS**: Follow existing Tailwind CSS patterns
- **Comments**: Add comments for complex logic
- **Naming**: Use descriptive variable and function names

#### Testing

Before submitting a Pull Request:

- [ ] Test in latest Chrome, Firefox, Safari
- [ ] Test in mobile view (responsive design)
- [ ] Test dark mode functionality
- [ ] Verify form validation works correctly
- [ ] Test with sample data in Google Sheets (if applicable)
- [ ] Ensure no console errors

## üìã Pull Request Process

1. **Update documentation** if you've changed functionality
2. **Add comments** to explain complex code
3. **Follow commit message conventions**:
   - `feat:` - New feature
   - `fix:` - Bug fix
   - `docs:` - Documentation changes
   - `style:` - Formatting changes
   - `refactor:` - Code restructuring
   - `test:` - Adding tests
   - `chore:` - Maintenance tasks

4. **Describe your changes** in the PR description:
   - What changed and why
   - Related issues (if any)
   - Testing performed
   - Screenshots (if UI changes)

5. **Wait for review** - maintainers will review and provide feedback

## ‚ùå What We Don't Accept

- Changes that break existing functionality without good reason

## üîê Security Issues

**Do NOT open public issues for security vulnerabilities.**

Instead, please email security concerns privately or open a security advisory through GitHub's private reporting feature.
